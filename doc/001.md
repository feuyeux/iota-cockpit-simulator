# Cockpit World Simulation Harness

## 1. 目的与依据

本文指导开发一套聚焦座舱仿真的独立 desktop app + simulation agent。它只复用本地参考项目 D:\coding\iota-sympantos 中的 iota-core 作为 Rust library；不复用或修改 iota-cli、iota-desktop、iota-kanban、iota daemon。

参考项目的已知事实：

- Rust workspace，最低 Rust 版本为 1.95.0。
- iota-core 提供可复用的 ACP 后端编排、Context、Memory、Skill、MCP、store 和 telemetry。
- iota-desktop 的技术栈是 Tauri 2 + React 19 + Vite 7 + TypeScript + Tailwind 4 + Lucide；本项目复用技术栈，不复用代码、UI、daemon 协议或产品模型。
- Agent 后端通过 ACP 接入 Codex、Claude、Gemini、Hermes、OpenCode 等。

iota 不是仿真引擎，也没有一个叫 itoa 的 Intent-Thought-Observation-Action 标准协议。本项目把仿真实现为独立 domain runtime；Agent integration 仅通过封装 iota-core 的 ACP、Context、Memory、Skill、MCP 与 telemetry API 完成。

## 2. 产品目标

用户在一台桌面电脑上可以：

1. 打开、校验和编辑版本化的 cockpit Scenario。
2. 启动、暂停、单步、停止、加速和回放一个确定性仿真 Run。
3. 通过 Sensor/MCP tool 让 Agent 看到 Perceived World，而不是 Ground Truth。
4. 让 Agent 通过受权限控制的 Action 改变世界。
5. 查看世界、事件、工具调用、Agent 输出、指标和错误。
6. 保存 recording，并在相同输入下复现或比较结果。

完整闭环：

~~~text
Cockpit desktop -> Tauri host -> RunManager -> simulation service
                -> sensor observation via local MCP -> iota-core ACP agent
                -> validated action -> world/event commit
                -> recording/replay/evaluation
~~~

### 2.1 MVP 垂直场景

必须先完成：

~~~text
Smoke enters cockpit
 -> visibility/noise changes
 -> sensor returns degraded observation
 -> Agent detects risk
 -> Agent calls engine shutdown or alarm action
 -> device/human state changes
 -> event, trace, metric and replay are available
~~~

### 2.2 不属于 MVP

云端分布式仿真、强化学习训练、自动 Scenario evolution、真实硬件闭环、多人编辑和完整 3D 渲染不阻塞第一版。架构预留接口，但不能先做这些功能再补基础闭环。

## 4. 核心不变量

1. Ground Truth 只由 simulation service 持有和提交；UI 与 Agent 不可直接写状态。
2. Agent 只能使用授权 Sensor/MCP tool 获取 Observation。
3. 所有世界修改经过 Action Gateway、schema、权限和前置条件校验。
4. 每个 tick 只提交一个不可变状态版本，提交后不得追写旧状态。
5. 相同 Scenario 版本、seed、配置、plugin hash 和输入序列必须得到相同 snapshot hash。
6. 仿真时间与墙上时间分离；Pause 时 sim time 不推进。
7. 事件按 scheduled_tick、priority、sequence 稳定排序，不依赖线程调度。
8. 工具调用、工具结果、拒绝、超时和异常进入 recording/trace。
9. prompt、API key、完整模型隐藏推理和其他 secret 不写入 recording 或 UI。
10. 不得依赖 iota-cli、iota-desktop、iota-kanban 或 iota daemon 的内部类型和协议；本项目的 Tauri/MCP/recording contract 必须独立版本化。

## 5. 运行模型

### 5.1 Run 生命周期

~~~text
Created -> Validating -> Ready -> Running -> Paused -> Running
                           |                 |
                           +-> Failed <------+
Running -> Completed / Stopped
~~~

Pause 在 tick 边界生效；Stop 结束后续动作但保留已有记录；Failed 需要新建 Run，不能从未知状态继续。

### 5.2 Tick 算法

~~~text
1. 读取 WorldSnapshot(tick)
2. 投递到期 scheduled events
3. 执行 Environment/Device/Human systems，产生候选变化
4. 执行 Influence，产生候选事件
5. Sensor 从 Ground Truth 生成延迟/噪声/丢包 observation
6. 通过 simulation MCP 暴露本 tick 的授权 observation
7. iota Agent 调用 observation/action tools
8. Action Gateway 校验并稳定排序 action
9. Commit StateDiff + EventBatch，生成 snapshot hash
10. 记录 tick，推进 sim time
~~~

Agent 或外部模型超时不得让内核死锁。使用 scenario 的 fallback policy，写入 AgentTimeout，并把 Run 标记为 degraded。

### 5.3 时钟模式

~~~text
realtime    tick 与墙上时间对齐
accelerated 尽快推进但遵守逻辑顺序
stepped     一个 Step 命令推进一个 tick
replay      不调用 Agent，使用原始 observation/action 输入
~~~

基础接口应提供 mode、tick、sim_time_ms、start、pause、step 和 stop。

## 6. World Model

采用 ECS 风格的数据组织；MVP 可以用 Rust struct + system 实现，但组件数据必须与系统逻辑分离。

~~~text
Entity: pilot-1
Components: Transform, HumanState, Attention, Goal, SensorRig
Systems: WeatherSystem, DeviceSystem, HumanSystem, SensorSystem
~~~

所有实体包含 id、type、tags、components、lifecycle 和 metadata。组件 schema 必须可序列化、可校验、带版本和单位。

### 6.1 Environment

MVP 实现 temperature、humidity、visibility、smoke、lighting、noise。变化来自 rule、scheduled event 或 plugin；数值必须有单位、范围、clamp 和来源。

### 6.2 Human

MVP 实现 stress、fatigue、health、attention、knowledge、memory、goal、location。Human 只能发出 speak、move、operate、request_help 等公开 command/event，不能直接修改设备。

### 6.3 Device

设备包含 health、power_state、faults、capabilities 和 lifecycle。通用状态为 normal -> warning -> failed -> recovering；每种设备必须定义自己的转换、前置条件和错误码，例如 DEVICE_UNPOWERED、PRECONDITION_FAILED、CAPABILITY_DENIED。

## 7. Scenario 格式

采用 YAML 或 JSON。加载时校验 schema、引用、单位、范围、权限和初始状态。

~~~yaml
schemaVersion: 1
id: smoke-in-cockpit
seed: 42
clock: { mode: stepped, tickMs: 100 }
entities:
  - id: cabin
    type: environment
    components: { smoke: { density: 0.0 } }
  - id: pilot-1
    type: human
    components: { attention: { value: 0.9 }, location: cockpit }
  - id: engine-1
    type: device
    components: { capabilities: [shutdown] }
faults:
  - atTick: 50
    target: cabin
    type: smoke_increase
agents:
  - id: cockpit-agent
    backend: codex
    observationProfile: pilot-default
    capabilities: [engine.shutdown, alarm.activate]
evaluation:
  - id: shutdown-before-spread
    deadlineTick: 300
    rule: engine.shutdown within 30 ticks after SmokeDetected
~~~

Scenario 必须有 content hash；recording 绑定 scenario hash，文件变化后不得假装兼容 replay。

## 8. Sensor 与 Perceived World

Agent 不允许调用 get_ground_truth。Sensor Profile 配置采样周期、延迟、噪声分布、丢包率、分辨率、遮挡规则和权限。

Observation 至少包含 observation_id、run_id、agent_id、sensor_id、observed_tick、delivered_tick、data、confidence 和 quality。

Ground Truth 与 Perceived World 必须分开存储和展示。UI 只能在明确调试权限下对照二者；Agent tool response 只能包含后者。视觉 MVP 可输出结构化目标列表，后续再增加图像。

## 9. Action Gateway

仿真提供 typed command，而不是任意 JSON patch。ActionRequest 至少包含 request_id、agent_id、target、command、args、expected_state_version 和 expires_at_tick。

处理顺序：

~~~text
schema -> agent identity -> capability -> target scope
      -> precondition -> expiry/version
      -> deterministic ordering -> execute
~~~

成功返回 ActionApplied；失败返回稳定错误码并发布 ActionRejected。Action 不得跨 tick 修改已经提交的 snapshot。

## 10. iota Agent 集成

### 10.1 MCP tools

首批工具：

~~~text
simulation.get_observation
simulation.list_visible_entities
simulation.inspect_sensor_quality
simulation.request_action
simulation.get_action_result
simulation.get_run_status
~~~

工具必须带 JSON schema、描述、权限和 side-effect 标记。get 类工具为 read-only；request_action 为 mutation，必须经过 Action Gateway。结果包含 runId、tick 和 correlationId。

MCP server 不能把 Rust World 引用暴露给 Agent。每次响应依据 Agent、Run、tick 和 capability 生成最小数据。

### 10.2 Skill

通过 iota Skill Registry 提供 cockpit-simulation skill，内容包括场景规则和单位、Agent 职责、Sensor 可信度和延迟、允许调用的 tools、fallback 行为和不得泄漏 Ground Truth 的规则。

Skill 是提示和工具编排层，不是安全边界；安全必须由 service/action gateway 强制执行。

### 10.3 ACP backend

Agent 后端仍由 iota-core ACP 管理，simulation service 不直接依赖某个模型，只等待 iota 的 tool call 和结果。

~~~text
ScriptedAgent -> 确定性集成测试
RuleAgent     -> 离线基线
ACP backend   -> codex/claude/gemini/hermes/opencode
~~~

外部 backend 必须有 timeout、cancel、并发上限、重试/断路器和 fallback。simulation tool 默认不允许网络访问；模型网络权限由 iota 配置显式决定。

## 11. Event Bus 与 Influence

事件 envelope 至少包含 event_id、event_type、run_id、tick、source、priority、correlation_id 和 payload。

Event Bus 支持发布、订阅、延迟投递、优先级和 recording。内部不承诺跨进程 exactly-once；消费者必须能检测重复。

Influence 声明 source -> target、权重、阈值、延迟和函数版本。它只产生候选事件或 StateDiff，最终由 system 在 commit 阶段应用；每条传播链必须能在 trace 中解释。

### 11.1 调度 Influence、仲裁与订阅（已实现）

- `cockpit_simulation_core::influence` 提供版本化、可调度的 `InfluenceRule`（`InfluenceSchedule::AtTick`/`Every`，`InfluenceOp::Set`/`Delta`，带 `rule_version` 与 `priority`）。Scenario 通过 `influences` 与 `conflictPolicy` 字段声明（均 `#[serde(default)]`，未配置时 tick 行为与 replay hash 完全不变）。
- 每个 tick：`schedule_due` 选出到期规则（按 entity/component/priority-desc/rule_id 全序），`arbitrate` 在同一 `(entity, component)` 冲突时按 `ConflictPolicy`（`RejectConflicting`/`HighestPriorityWins`/`LowestRuleIdWins`）确定性裁决。胜出规则复用与外部 StateDiff 相同的范围校验与写入路径，在 commit 阶段发出 `InfluenceApplied`；落败/越界规则发出 `InfluenceRejected` 证据事件。
- 订阅：`Subscription`（按 `event_types` 过滤，空表示全部）提供确定性 `deliver` 选择器，作为可复用原语。
- 覆盖：`crates/cockpit-simulation-core/src/influence.rs` 单元测试 + `tests/contract/influence_rules.rs`（应用、拒绝、确定性仲裁、无规则不变）。

## 12. Recording、Replay、Evaluation

每个 Run 记录 scenario content hash、应用/schema/plugin 版本、seed、clock 配置、events、observations、tool calls、actions、results、errors、fallback、每 tick snapshot hash 和 evaluation。

Replay 禁用外部 Agent，使用原始 observation/action 输入。版本不兼容时明确拒绝。Diff 支持 tick、entity、component path、event、tool call 和 metric。

Evaluator 示例：

~~~text
WHEN event SmokeDetected
THEN engine.shutdown occurs within 30 ticks
AND no unauthorized action occurs
~~~

结果包含 passed、score、evidence_event_ids、first_failure_tick 和 explanation。

## 13. Desktop 规格

新建 cockpit-desktop，采用与参考项目相同的 Tauri 2 + React 19 + Vite 7 + TypeScript + Tailwind 4 + Lucide 技术栈。它不复用 iota-desktop 的组件、reducer、Tauri command 或 daemon-first 协议，也不在前端运行仿真内核。

~~~text
+-----------------------------------------------------------+
| project/scenario | Start Pause Step Stop | tick/sim time  |
+------------------+----------------------+----------------+
| entity tree      | 2D cockpit/world view | inspector      |
| filters          | timeline overlays    | component state|
+------------------+----------------------+----------------+
| events | agent trace/tool calls | sensor | metrics      |
+-----------------------------------------------------------+
~~~

面板：

- Scenario：文件、版本、seed、schema errors、Agent 配置。
- Run Control：状态、tick、sim time、speed、Start/Pause/Step/Stop。
- World View：实体树、筛选、二维布局和最近变化。
- Inspector：组件、来源事件、Sensor quality、可用 action。
- Timeline：事件、Observation、tool call、Action 对齐。
- Agent Trace：backend、session、skill、tool args/result、latency、错误。
- Evaluation：指标、deadline、证据和 replay diff。

必须实现 loading、empty、no Agent、paused、running、failed、service disconnected 和 reconnect 状态。Ground Truth 面板默认隐藏；窗口缩放、键盘 Step/Pause 和大 JSON 不能破坏布局。

## 14. Desktop/runner IPC contract

cockpit-desktop 与本项目的 cockpit-runner 使用独立的 versioned tagged message contract。Tauri host 负责启动、认证、停止 runner，并将 runner 事件转发给 React；不得复用 iota daemon 或 ACP wire message：

~~~text
ValidateScenario
CreateSimulationRun
StartSimulation
PauseSimulation
StepSimulation
StopSimulation
GetSimulationSnapshot
GetSimulationEvents
GetAgentTrace
StartReplay
~~~

runner 推送：

~~~text
SimulationStateChanged
SimulationTickCommitted
SimulationEvent
SimulationToolCall
SimulationActionResult
SimulationEvaluationUpdated
SimulationError
~~~

UI 只提交 command，不提交 arbitrary state patch。错误统一包含 code、message、details、runId、tick 和 correlationId。runner 重连必须用 snapshot + event cursor 恢复，而不是只刷新 UI。IPC 默认只监听 loopback，并以 Tauri host 生成的短期 session token 鉴权。

## 15. 存储、安全与插件

Scenario 使用项目文件；Run metadata、event stream、recording index 和 evaluation 使用 SQLite。大 payload 使用 content-addressed 文件，SQLite 保存 hash。

不得存储 API key、完整 secret-bearing config、隐藏推理。日志字段包含 runId、tick、component、event_type、agent_id、correlation_id，并自动 redact。

插件 manifest 声明 id、version、API contract、权限、schema 和 hash。插件不能绕过 Action Gateway；权限未知、版本冲突或 hash 不匹配时拒绝加载。

## 17. 分阶段交付

### Phase 0：合同与 cockpit 内核垂直切片

建立 cockpit-simulation-core、Scenario schema、错误码、seeded RNG、ScriptedAgent 和 smoke scenario，完成无 UI 的 load -> tick -> observation -> action -> commit -> record -> replay 测试。

### Phase 1：MCP、iota-core adapter 与 runner

实现本项目的 MCP tools、Skill adapter、capability enforcement、timeout/fallback 和 cockpit-runner IPC contract。用 RuleAgent 做离线演示。

### Phase 2：desktop MVP

在新建的 cockpit-desktop 中实现 Scenario、Run、World、Timeline、Trace、Evaluation 面板。每个按钮连接真实 cockpit-runner，不使用只改变前端的假状态。

### Phase 3：ACP Agent 与扩展

接入 Codex 等 ACP backend，完成 approval、cancel、断连恢复、memory/context 注入和多 Agent；之后再做 plugin、3D 和分布式 runner。

## 18. 测试与验收

### 18.1 自动化测试

- clock 的 pause/step/realtime 边界；
- event 延迟、优先级和重复检测；
- seed、noise、dropout 和 snapshot hash；
- Scenario/schema/unit/range/capability 校验；
- Action 前置条件、过期和 optimistic version；
- MCP response 不包含 Ground Truth；
- unauthorized action 一律拒绝；
- Agent timeout 使用 fallback；
- recording/replay 不调用 ACP 且 hash/metrics 一致；
- daemon disconnect/reconnect 的 snapshot + event cursor；
- cockpit 工作区的 Rust tests、fmt、clippy 与 cockpit-desktop build 通过；iota-core 以锁定 revision 的独立依赖进行兼容测试。

### 18.2 MVP 验收

1. 新开发者可按 README 启动 cockpit-runner 和新建的 Tauri desktop，并运行 smoke scenario。
2. 固定 seed 连续两次运行得到相同 final snapshot hash 和 evaluation。
3. UI 可见 event、sensor quality、iota tool call、action result、错误和指标。
4. Agent 无法读取 Ground Truth；越权 action 有 ActionRejected 和证据。
5. Pause 不推进 sim time；Step 恰好推进一个 tick。
6. Replay 不启动外部模型，结果与原始 run 一致。
7. UI、daemon、simulation service 任一异常都有可见状态和结构化错误。
8. 在记录硬件和配置的前提下达到 1000 entities、10,000 events/min、tick p95 < 50ms。
9. 跨平台打包和测试通过，日志和截图不暴露 secrets。

推荐命令（在 cockpit-simulation 根目录）：

~~~bash
cargo fmt --all --check
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
cargo run -p cockpit-runner -- validate scenarios/smoke-in-cockpit.yaml
cd apps/cockpit-desktop && npm test && npm run build
~~~

## 19. Definition of Done

功能只有同时拥有 schema/contract、Rust service 实现、MCP/Skill 或 daemon 接入、desktop 状态、错误语义、record/replay 行为、自动化测试和可运行示例，才算完成。增加一个按钮、一次模型调用或一组 mock 数据不算完成仿真能力。

## 20. ADR 待确认项

实现前需要确认：

- simulation service 是独立进程还是暂时嵌入 daemon；
- recording 的 SQLite 表结构和大 payload 文件格式；
- desktop protocol 扩展现有 TCP contract 还是使用独立 WS；
- Sensor 噪声分布和图像格式；
- plugin sandbox 级别；
- 生产环境允许哪些 ACP backend 和网络权限。

任何改变 iota-core adapter、ACP、MCP、cockpit-runner IPC 或 replay 兼容性的决策，都必须同步更新 contract version、迁移策略和验收测试。

## 21. 现有 iota 模块的具体落点

以下映射以 iota-sympantos 的 iota-core public API 和 desktop 技术栈为参考。除 iota-core library dependency 外，所有文件均创建在本项目中：

| 仿真能力 | 首选落点 | 责任 |
| :--- | :--- | :--- |

| Scenario 解析与校验 | crates/cockpit-scenario | YAML/JSON、版本和 hash |
| MCP tool server/dispatch | crates/cockpit-agent-runtime | 调用 iota-core public MCP API，提供最小 observation |
| Agent skill/context | crates/cockpit-agent-runtime | 调用 iota-core public Skill/Context/Memory API，保留 cockpit 规则所有权 |
| ACP turn/session | crates/cockpit-agent-runtime | 调用 iota-core public ACP/Engine API，不复制后端进程管理 |
| runner 命令和流事件 | crates/cockpit-runner | 本项目 IPC、cursor、重连 |
| recording/index | crates/cockpit-recording | typed SQLite operations |
| tracing/metrics | crates/cockpit-agent-runtime、cockpit-runner | 复用 iota-core telemetry public API |

| 项目任务编排 | 本项目自行决定 | 不依赖 iota-kanban |

依赖规则必须保持：store 不调用 UI/ACP/MCP client，presentation 不拥有 ACP session，cockpit-simulation-core 不依赖 presentation，iota-core 不能被本项目修改或反向引用。

## 22. 一次 Agent 工具调用的实际链路

开发和调试必须能沿以下链路定位问题：

~~~text
React Run view
 -> cockpit Tauri command
 -> Tauri host starts/authenticates cockpit-runner
 -> cockpit-runner obtains SimulationRunHandle
 -> cockpit-simulation-core commits tick
 -> cockpit-agent-runtime serves get_observation
 -> iota-core public MCP/Engine API dispatches typed tool
 -> ACP session sends tool description/result to backend
 -> backend emits tool call
 -> cockpit-agent-runtime invokes request_action
 -> ActionGateway validates capability and precondition
 -> cockpit-simulation-core commits ActionResult
 -> RuntimeEvent + recording + runner stream
 -> Tauri event + React reducer update timeline/inspector
~~~

每一跳都必须携带 run_id、tick 和 correlation_id。出现问题时，不能只在 React 控制台打印一个字符串；必须能从 desktop trace 反查到 service event 和 recording。

## 25. 本地开发与手工验收 Runbook

参考 iota-sympantos 现有 desktop acceptance runbook，新增仿真验收步骤。所有日志、截图和 issue 内容不得包含 API key、token、原始 nimia.yaml 或 secret-bearing payload。

### 25.1 启动

~~~bash
rustup default 1.95.0
cargo check --workspace
cargo test --workspace
cargo run -p cockpit-runner -- check-agent
~~~

启动 cockpit-runner 后，再从 apps/cockpit-desktop 启动新的 Tauri development mode。Windows 下 npm/npx 子进程应使用平台解析后的命令，不能硬编码 Unix shell。

### 25.2 Scenario

打开 smoke scenario，确认 schema、entity reference、capability 和 seed 均通过。故意把 device id 改错，确认 Run 按钮保持 disabled，并展示字段路径和错误码。

### 25.3 Deterministic run

使用 stepped 模式执行 10 个 tick：

- Pause 后 tick 和 sim time 不变；
- Step 一次只增加一个 tick；
- 同 seed、同 Scenario、同 ScriptedAgent 的两次运行 final snapshot hash 相同；
- recording 中不存在 API key 或完整 prompt secret。

### 25.4 Perception/action boundary

用 Agent trace 检查 Agent 只能看到 Sensor 输出。直接尝试调用 get_ground_truth 和无 capability 的 engine command，预期分别得到 TOOL_NOT_ALLOWED 或 CAPABILITY_DENIED，并产生可追踪事件。

### 25.5 Replay/disconnect

停止 ACP backend 后，Replay 仍应完成且不启动外部模型。运行中断开 desktop，再恢复连接，UI 应先收到 snapshot，再从 event cursor 继续，不能重复显示或跳过已提交 tick。

### 25.6 记录结果

每次验收记录 date、OS、iota commit、Scenario hash、backend、seed、result 和 notes。没有实际运行记录时只能称为“未验证”，不能把代码存在当作验收通过。

## 26. Domain contract 草案

以下类型是 cockpit-simulation-core 的最小公共合同。字段名一旦用于 recording、MCP 或 runner IPC，就必须通过 schema version 演进，不得随意重命名。

~~~rust
pub type Tick = u64;
pub type StateVersion = u64;

pub struct RunId(pub String);
pub struct EntityId(pub String);
pub struct AgentId(pub String);

pub struct WorldSnapshot {
    pub run_id: RunId,
    pub tick: Tick,
    pub sim_time_ms: u64,
    pub state_version: StateVersion,
    pub entities: Vec<EntitySnapshot>,
    pub hash: String,
}

pub struct EntitySnapshot {
    pub id: EntityId,
    pub kind: String,
    pub components: serde_json::Value,
}

pub enum RunStatus {
    Created,
    Validating,
    Ready,
    Running,
    Paused,
    Completed,
    Stopped,
    Failed,
}

pub struct StateDiff {
    pub run_id: RunId,
    pub tick: Tick,
    pub base_version: StateVersion,
    pub operations: Vec<StateOperation>,
}
~~~

StateOperation 只允许明确的 component/path 操作，例如 set_component、remove_component、emit_event 和 schedule_event；禁止暴露 set_world 或任意 pointer patch。提交失败必须返回 base version mismatch，而不是自动覆盖并继续运行。

## 27. MCP tool contract 草案

### 27.1 get_observation

输入只允许：

~~~json
{
  "run_id": "run-001",
  "agent_id": "cockpit-agent",
  "sensor_id": "pilot-vision",
  "max_age_ticks": 3
}
~~~

输出必须是 Perceived World：

~~~json
{
  "run_id": "run-001",
  "tick": 64,
  "observation_id": "obs-64-01",
  "objects": [
    {
      "id": "engine-1",
      "kind": "engine",
      "state": "uncertain",
      "confidence": 0.72
    }
  ],
  "quality": {
    "latency_ticks": 2,
    "dropped": false,
    "occluded": true
  },
  "correlation_id": "corr-001"
}
~~~

### 27.2 request_action

输入必须包含 target、command、args、expires_at_tick 和 correlation_id。服务端不得接受 world patch、component replacement 或未注册 command。

~~~json
{
  "run_id": "run-001",
  "agent_id": "cockpit-agent",
  "target": "engine-1",
  "command": "shutdown",
  "args": { "reason": "smoke_detected" },
  "expires_at_tick": 70,

}
~~~

输出统一使用：

~~~json
{
  "request_id": "req-001",
  "status": "applied",
  "applied_tick": 65,
  "new_state_version": 65,
  "error": null,
  "correlation_id": "corr-002"
}
~~~

拒绝时 status 为 rejected，error 必须包含稳定 code。错误 code 至少包括 INVALID_SCHEMA、RUN_NOT_ACTIVE、CAPABILITY_DENIED、TARGET_NOT_VISIBLE、PRECONDITION_FAILED、ACTION_EXPIRED、STATE_VERSION_CONFLICT 和 INTERNAL_ERROR。

## 28. Tick 提交与并发规则

MVP 使用单一 simulation executor，所有 systems 按固定顺序执行。Agent tool call 可以异步到达，但不能直接修改 World。每个 tick 维护三个集合：

~~~text
candidate_events  systems 产生但尚未提交的事件
pending_actions   Gateway 接收但尚未提交的动作
commit_batch      本 tick 最终排序并应用的变化
~~~

提交规则：

1. 读取 tick 开始时的 snapshot。
2. systems 只能读取 snapshot，并写入 candidate_events。
3. tool call 只能产生 pending_actions。
4. 按 priority、agent_id、request_id 对 pending_actions 稳定排序。
5. 对每个 action 校验 capability、可见性、前置条件、版本和过期时间。
6. 合法 action 转成 StateOperation；拒绝 action 只写 event，不改变 World。
7. 一次性 commit StateDiff、events 和 hash。
8. commit 完成后才向 desktop 推送 TickCommitted。

这样可以避免“UI 看到动作成功但 World 尚未提交”或“两个 Agent 依赖线程先后顺序产生不同结果”。

## 29. Agent 失败与降级矩阵

| 故障 | 内核行为 | UI 状态 | recording |
| :--- | :--- | :--- | :--- |
| ACP backend 未启动 | 使用 RuleAgent 或 scenario fallback | degraded | AgentUnavailable |
| tool call 超时 | 拒绝本次调用，不重放旧动作 | warning | ToolTimeout |
| Agent 输出非法 schema | 丢弃输出 | error on trace | InvalidAgentOutput |
| Action 无权限 | 不改状态 | rejected | ActionRejected |
| Sensor 丢包 | 返回 dropped quality | degraded sensor | SensorDrop |
| recording 写失败 | 按 policy pause 或 fail | recording error | RecordingFailure |
| cockpit-runner 断开 | simulation service 继续运行 | reconnecting | connection events |
| service panic/退出 | Run failed | service unavailable | terminal failure |

fallback 不能静默改变评测语义。Evaluator 必须知道某个 tick 是否由 fallback 产生，并可将 Run 标记为 invalid、degraded 或 valid。

## 31. 当前明确的实现假设

在用户未另行指定前，按以下假设推进：

- 使用 Rust 实现 cockpit-simulation-core，并建立独立 cockpit-simulation workspace。
- 初期 simulation service 独立进程，避免 Agent/ACP 故障拖垮 kernel。
- 桌面通信使用本项目的 cockpit-runner IPC，由 Tauri host 管理，不使用 iota daemon。
- MCP 作为 Agent 与仿真的唯一工具边界。
- SQLite 保存 metadata/event index，文件系统保存大 recording payload。
- MVP 使用结构化 observation，不要求真实视觉模型。
- ScriptedAgent 和 RuleAgent 必须先于外部 ACP Agent 完成。
- iota-core 通过 path dependency 开发、通过 git revision/release artifact 交付；禁止修改参考仓库源码作为本项目功能的一部分。

如果后续需要改变这些假设，必须同时更新第 4、10、14、18、30 和 31 节，并增加 ADR，而不是只改实现。

## 32. 版本、迁移和兼容性

仿真有三类版本，必须分别记录：

| 版本 | 作用 | 变化策略 |
| :--- | :--- | :--- |
| scenario schema version | Scenario 文件结构 | 读取器可提供显式 migration |
| runtime contract version | MCP、runner IPC、recording DTO | 只允许向后兼容增加字段，破坏性变化升主版本 |
| world model version | component 和 system 语义 | 不能仅靠 JSON schema 判断，需要 migration/拒绝 |

Recording header 至少包含 scenario_schema_version、runtime_contract_version、world_model_version、application_commit、plugin_hashes 和 seed。

迁移规则：

1. 新版本读取旧 Scenario 前先生成迁移后的内存副本，原文件不被静默修改。
2. 旧 recording 只能在明确兼容的 runtime/world model 下 replay。
3. 不能迁移的 recording 必须返回 INCOMPATIBLE_RECORDING，并列出冲突版本。
4. cockpit-desktop 与 cockpit-runner 的 contract version 不匹配时，握手阶段失败，不进入半连接状态。
5. MCP tool schema 变化必须增加 tool schema version，并在 trace 中记录实际版本。

### 32.1 迁移工具（已实现）

- `cockpit_recording::migrate` 提供版本常量（`CURRENT_SCHEMA_VERSION` 等）、链式迁移框架与 `migrate_recording_bytes`/`migrate_recording_value`。迁移在 `serde_json::Value` 层进行（旧 recording 缺字段无法直接反序列化到强类型），补齐 v0→v1 的 schema/runtime/world-model 版本与 provenance 默认值，且不覆盖已存在的值。
- 比当前 schema 更新的 recording 返回 `MigrationError::TooNew`，未知中间版本返回 `Unsupported`——即「不能迁移的 recording 明确拒绝并列出冲突版本」，而非静默接受。
- 工具入口：`cockpit-runner migrate-recording <input> [--output <path>] [--dry-run]`，输出 from/to 版本与已应用步骤。
- 覆盖测试：`tests/contract/recording_migration.rs`（迁移后可 replay 且 final snapshot hash 一致）+ `crates/cockpit-recording/src/migrate.rs` 单元测试。

## 33. 权限和威胁模型

信任边界如下：

~~~text
untrusted: ACP backend, external Agent, plugin, desktop renderer
trusted:   simulation service, Action Gateway, recording writer
restricted: Ground Truth, secrets, host process and filesystem
~~~

必须防御：

- Agent 通过 prompt/tool 参数读取未授权 entity 或 Ground Truth；
- Agent 伪造 agent_id、run_id、capability 或 state version；
- plugin 直接访问内核状态、文件系统或网络；
- 恶意 Scenario 使用超大数组、递归引用或无界 event schedule；
- desktop renderer 发送任意 state patch；
- recording 中写入 prompt、token、环境变量或 API key；
- ACP backend 返回超大 tool 参数、非法 JSON 或重复 request_id。

最低措施：

- capability 在 service 端绑定 agent、run、target scope 和 command；
- 所有 ID 在 service 端重新解析，不能信任客户端声明；
- Scenario 解析设置大小、深度、实体数、event 数和运行时长限制；
- plugin 默认无网络、无任意文件系统权限；
- tool 和 daemon payload 设置最大字节数；
- request_id、event_id 和 correlation_id 必须唯一并做重放检测；
- secret redact 在日志、trace、recording 和导出路径各自执行；
- desktop 与 service 的本地通信默认绑定 loopback，并校验 session token。

安全测试必须从攻击输入验证拒绝行为，而不能只检查正常用户流程。

## 34. 性能预算与容量策略

MVP 预算按单机、单 Run、单 simulation executor 计算：

| 指标 | 目标 | 超限行为 |
| :--- | :--- | :--- |
| tick p95 | 小于 50 ms | 记录 over-budget，不偷偷改变逻辑时间 |
| active entities | 1000 | Validate 阶段提示容量风险 |
| events | 10,000/min | queue depth 和 drop policy 可见 |
| observation payload | 1 MB/tool response | 拒绝或分页，不截断伪造数据 |
| recording queue | 有界 | pause 或 fail，不能无限增长 |
| Agent tool deadline | scenario 配置 | timeout + fallback |
| desktop stream | 有界 cursor | snapshot + cursor 恢复 |

benchmark 必须固定 OS、CPU、内存、Rust profile、Scenario hash、seed、entity 数和 event 数。报告同时给出平均值、p50、p95、p99、峰值内存和 recording 大小。

性能优化不能改变 deterministic ordering。并行 system 只能在明确的读写集合和固定合并规则下启用；MVP 默认串行。

## 36. 插件生命周期

插件加载顺序固定为：

~~~text
discover -> manifest parse -> signature/hash check
        -> API version check -> permission check
        -> schema registration -> isolated initialize
        -> Ready -> tick participation -> shutdown
~~~

插件不得在 initialize 阶段启动未声明的线程、网络或子进程。插件 tick 超时、panic 或返回非法 StateDiff 时：

1. 丢弃该插件本 tick 的候选变化；
2. 发布 PluginFailure；
3. 按 scenario policy disable plugin、pause Run 或 fail Run；
4. 在 recording 中写入 plugin version、错误和处理决策。

插件输出必须经过和 Agent Action 相同的 StateDiff 校验，不能因为是“内部插件”就绕过权限和范围检查。

### 36.1 Sandbox 级别（ADR §20「plugin sandbox 级别」已确认）

当前实现的 sandbox 级别为 **进程内合作式 (in-process cooperative)**：

- 插件通过 `PluginExecutor` trait 在宿主进程内同步执行；宿主对每个 tick 施加合作式 wall-clock 预算 `PluginPolicy.tick_budget_ms`（默认 50ms）。返回超过预算的插件按 `failure_policy` 失败处理（丢弃候选变化、发布 `PluginFailure`、disable/pause/fail）。
- 该预算能约束「返回但过慢」的插件，但**不能抢占已挂起或 panic 的插件**——真正的 OS 级抢占、内存/CPU 配额与文件系统/网络隔离需要**进程外执行**（子进程或 wasm 沙箱），列为后续 Phase 工作。
- 在进程外执行落地前，宿主**只运行受信任的内建 executor**，不加载第三方插件二进制；所有插件输出仍然经过完整的 manifest/permission/scope/version/StateDiff 校验（fail-closed）。

## 40. 仿真元素交互模型

本节定义座舱世界中各元素如何相互影响。它解决的不是“谁调用谁”的代码组织问题，而是“一个物理或行为变化在什么时间、以什么条件、影响哪些对象、何时被 Agent 看见、如何被动作改变”的世界模型问题。

### 40.1 两条必须分离的链路

每一次变化都同时可能经过两条链路：

1. 因果链路：Ground Truth 中环境、人、设备之间的真实影响。
2. 信息链路：Sensor 将部分 Ground Truth 变成有噪声、有延迟、有权限限制的 Observation。

Agent 只位于信息链路末端，不能跳过 Sensor 读取因果链路中的任意状态。UI 也不参与任一因果链路，它只订阅已提交的 snapshot、event 和 trace。

~~~mermaid
flowchart LR
    E[Environment smoke wind lighting] --> P[Physical world state]
    D[Device engine power alarm] --> P
    H[Human stress attention location] --> P
    P --> I[Influence systems]
    I --> P
    P --> S[Sensor profiles latency noise occlusion]
    S --> O[Perceived World Observation]
    O --> A[iota-core Agent runtime]
    A --> G[Action Gateway]
    G --> P
    P -. committed snapshots only .-> U[Desktop UI]
    O -. trace only .-> U
    G -. action result .-> U
~~~

原理：环境、设备和人共同构成世界状态；Influence system 只根据上一个已提交 snapshot 计算候选变化；Sensor 负责信息损失；Action Gateway 是 Agent 从信息世界返回物理世界的唯一入口。这使“烟已经变浓，但 Agent 还没看见”成为可表达、可回放的正常状态。

### 40.2 元素职责与交互边界

| 元素 | 读取 | 输出 | 典型影响 | 禁止行为 |
| :--- | :--- | :--- | :--- | :--- |
| Environment system | 环境组件、到期事件 | 环境 StateDiff、环境事件 | smoke 降低 visibility，temperature 影响 human health | 直接替 Agent 下动作 |
| Device system | 自身状态、power、环境、合法 action | 设备 StateDiff、warning/failure event | engine fire 产生 heat/smoke，power loss 关闭 display | 直接修改 Human 的意图 |
| Human system | 可达环境、设备状态、内部心理状态、事件 | stress/attention/location 变化、人类行为 intent | alarm 增加 stress，低 attention 增加操作失误率 | 直接读取 Agent 的隐藏上下文 |
| Influence system | 已提交 snapshot、规则参数 | 候选 StateDiff/Event | smoke -> visibility，alarm -> stress，power -> display | 直接 commit 或依赖迭代顺序 |
| Sensor system | Ground Truth、Sensor Profile | Observation | camera 受烟遮挡，microphone 受噪声影响 | 返回未授权 Ground Truth |
| Agent runtime | Observation、ActionResult、授权 skill/context | tool call、ActionRequest | 判断风险、请求关停引擎、询问状态 | 写入 World 或修改 Sensor |
| Action Gateway | ActionRequest、capability、snapshot | ActionApplied/Rejected、候选操作 | 将 shutdown 映射为设备命令 | 依赖模型自觉守规 |
| Event Bus | 已提交事件、订阅规则 | 到期 event delivery | 将 FireDetected 广播给 alarm/human/sensor | 直接修改组件 |
| Desktop UI | committed snapshot、event、trace | 用户 command | pause、step、选择实体 | 直接 patch World |

### 40.3 同 tick 内的因果原则

一个 tick 采用“读取旧快照、计算候选变化、一次提交新快照”的模型。任何 system 不得读取另一个 system 在同 tick 的未提交写入；否则 system 注册顺序会改变结果，回放将不再确定。

~~~mermaid
flowchart TD
    S0[Snapshot tick N] --> EV[Deliver scheduled events]
    EV --> ENV[Environment system]
    EV --> DEV[Device system]
    EV --> HUM[Human system]
    ENV --> C[Candidate changes]
    DEV --> C
    HUM --> C
    C --> INF[Influence calculation]
    INF --> B[Commit batch]
    ACT[Validated actions for tick N] --> B
    B --> S1[Snapshot tick N plus 1]
    S1 --> REC[Recording hash and events]
    S1 --> UI[Desktop stream]
~~~

原理：

- Environment、Device、Human 可并列读取 Snapshot N，但 MVP 仍按固定顺序执行，保证简单且可复现。
- Influence 不直接修改对象，而是把 smoke、heat、power、alarm 等跨域作用转成可审计的候选变化。
- Action 的生效点是 commit batch；因此 UI 只有在收到 ActionApplied 对应的 committed tick 后才能显示“已生效”。
- Sensor 对 Snapshot N 或由 Scenario 指定的历史 snapshot 采样；它不能偷看 Snapshot N plus 1。

### 40.4 环境、设备与人的物理和行为耦合

交互规则应声明为有方向、可计算、可解释的边，而不是散落在任意 handler 中。每条边定义 source component、target component、函数、阈值、延迟、上限和证据事件。

~~~mermaid
flowchart LR
    FIRE[Engine fire] -->|heat rate| TEMP[Cabin temperature]
    FIRE -->|smoke rate| SMOKE[Cabin smoke density]
    SMOKE -->|attenuation| VIS[Visibility]
    SMOKE -->|irritant dose| HEALTH[Human health]
    VIS -->|occlusion| CAM[Camera observation quality]
    ALARM[Alarm active] -->|noise level| NOISE[Cabin noise]
    NOISE -->|speech intelligibility| MIC[Microphone quality]
    ALARM -->|stress increment| STRESS[Human stress]
    STRESS -->|attention penalty| ATTN[Human attention]
    ATTN -->|error probability| HUMANACT[Human operation outcome]
    POWER[Power bus] -->|supply state| DISPLAY[Display availability]
    HUMANACT -->|operate switch| POWER
~~~

这张图中的边不是都要在同一个 tick 生效：

| 边 | 计算方式 | 推荐延迟 | 结果 |
| :--- | :--- | :--- | :--- |
| engine fire -> smoke | 每 tick 增加 density，受 ventilation 抵消 | 0 tick | SmokeDensityChanged |
| smoke -> visibility | visibility = base * attenuation(density) | 0 tick | VisibilityChanged |
| smoke -> health | 按浓度与暴露时间累计 dose | 1 tick 或以上 | HealthChanged |
| alarm -> stress | 阈值触发后以速率增加，受 training/fatigue 调制 | 1 tick | StressChanged |
| stress -> attention | 非线性衰减并 clamp 到 0..1 | 1 tick | AttentionChanged |
| attention -> operation outcome | 以 seeded RNG 采样失败概率 | action commit 时 | ActionApplied 或 ActionFailed |
| power -> display | power state 立即决定 display 可用性 | 0 tick | DeviceUnavailable |

同一条边的函数必须纯粹且带 version。例如 visibility_attenuation_v1 的输入只有 smoke density 和配置；随机性必须来自 Run 的 seeded RNG，并把 RNG draw 记录到 recording。

### 40.5 感知、决策、动作与反馈

Sensor 不是“世界状态的 JSON 序列化器”，而是独立的观测模型。不同 Actor 可以同时看向同一设备，却得到不同 Observation：camera 受烟遮挡，microphone 受噪声干扰，经过训练的 pilot 可从 warning tone 推断故障，普通 passenger 不具备该知识。

~~~mermaid
sequenceDiagram
    participant W as Ground Truth
    participant S as Sensor system
    participant M as Local MCP boundary
    participant A as Agent runtime
    participant G as Action Gateway
    participant D as Device system
    participant R as Recording
    W->>S: sample at observed tick
    S->>S: apply latency noise occlusion permission
    S->>M: Observation with quality metadata
    M->>A: authorized perceived world only
    A->>M: request action shutdown engine-1
    M->>G: typed ActionRequest
    G->>G: schema capability state expiry checks
    alt accepted
        G->>D: candidate shutdown operation
        D->>W: commit device state at tick boundary
        G->>A: ActionApplied
    else rejected
        G->>A: ActionRejected with stable error code
    end
    W->>R: snapshot hash event and result
~~~

原理：

1. observed_tick 与 delivered_tick 可以不同。Agent 收到的警报可能来自两个 tick 前的世界。
2. ActionRequest 附带 expected_state_version 和 expires_at_tick；因而 Agent 根据旧 Observation 发出的动作可能被拒绝，这不是异常，而是并发与延迟的真实表现。
3. Agent 的文本解释只用于 trace，不能作为 world transition 的条件。world transition 只由 typed action、规则和 deterministic system 决定。
4. ActionResult 必须成为下一轮 Observation 或 context 的一部分，否则 Agent 无法区分“动作成功”“动作被拒绝”“传感器暂时没更新”。

### 40.6 事件传播与延迟

Event 表示已发生或将发生的离散事实；StateDiff 表示连续或持久状态的改变。二者必须同时存在。

~~~mermaid
flowchart TD
    F[EngineFire event at tick 50] --> A[Alarm subscriber]
    F --> H[Human stress subscriber]
    F --> L[Lighting subscriber]
    F --> E[Evaluator subscriber]
    A -->|schedule tick 51| AE[AlarmActivated event]
    AE --> N[Noise component change]
    AE --> S[Stress influence candidate]
    L -->|schedule tick 51| LE[EmergencyLighting event]
    S -->|commit tick 51| HS[HumanState stress change]
    E -->|open evaluation window| EV[Shutdown deadline starts]
~~~

事件优先级只决定同一 scheduled tick 的处理顺序，不应偷偷改变物理因果。延迟必须由事件的 scheduled_tick 或影响边的 delay 明确表示。重试仅适用于可靠投递或外部副作用，不能对已提交的世界变化重复执行。

### 40.7 多参与者动作冲突

冲突不是简单的“最后写入获胜”。例如 pilot、copilot 和 Agent 在同一 tick 对 engine-1 发出不同命令时，系统必须先归一化为设备级 command，再用明确规则裁决。

~~~mermaid
flowchart TD
    P[Pilot manual action] --> Q[Pending actions]
    C[Copilot action] --> Q
    A[Agent action] --> Q
    Q --> V[Validate authority target precondition expiry]
    V --> O[Stable order priority actor policy request id]
    O --> X{Commands compatible}
    X -->|yes| B[Build one commit batch]
    X -->|no| R[Apply configured conflict policy]
    R --> M[Manual override wins]
    R --> S[Safety shutdown wins]
    R --> J[Reject lower priority action]
    B --> E[Emit action result events]
    M --> E
    S --> E
    J --> E
~~~

MVP 的 conflict policy 必须写入 Scenario：

- safety command 优先于 convenience command；
- 已授权的 manual pilot command 优先于 Agent command；
- 同一 Actor 的相反命令以较新的、未过期 request 为准；
- 同一优先级、不可兼容命令按 request_id 稳定排序，后者得到 ACTION_CONFLICT；
- 任何裁决都产生 ActionApplied、ActionSuperseded 或 ActionRejected 事件。

禁止以模型输出的自然语言、到达线程的先后顺序或 UI 点击时间作为冲突裁决依据。

### 40.8 烟雾座舱场景的端到端解释

下面的示例是 MVP 的参考因果链。它描述的是同一 Run 中可以被 recording 和 replay 逐 tick 验证的状态变化：

~~~mermaid
flowchart LR
    T50[Tick 50 fault inject] --> FIRE[engine fire]
    FIRE --> SMOKE[smoke density rises]
    SMOKE --> VIS[visibility falls]
    SMOKE --> CAMQ[camera quality falls]
    FIRE --> EVT[EngineFire event]
    EVT --> ALARM[alarm activates]
    ALARM --> STRESS[pilot stress rises]
    VIS --> OBS[degraded visual observation]
    CAMQ --> OBS
    OBS --> AGENT[Agent chooses shutdown]
    AGENT --> GATE[Action Gateway]
    GATE --> SHUT[engine shutdown]
    SHUT --> FIREOFF[fire source stops]
    FIREOFF --> DECAY[smoke decays by ventilation]
    SHUT --> EVAL[evaluator checks deadline]
~~~

在这个场景中，Agent 不会直接看到 smoke density 或 fire source；它只会看到低可信度的视觉 Observation、可能被噪声覆盖的 alarm、可见设备状态和此前 action result。若 Agent 在 deadline 前成功 shutdown，fire source 停止，但 smoke 不应瞬间归零，仍需由 ventilation 规则逐 tick 衰减。这种“动作解决根因但不立即恢复表象”的设计是座舱仿真可信度的关键。

### 40.9 交互规则的配置和可测试性

每条影响边、事件订阅和 conflict policy 必须能从 Scenario 或版本化 plugin 配置追溯。至少记录：

~~~yaml
id: smoke-reduces-visibility
source: cabin.smoke.density
target: cabin.visibility
function: visibility_attenuation_v1
delayTicks: 0
threshold: 0.05
clamp: [0.0, 1.0]
emits: VisibilityChanged
~~~

交互测试至少包含：

- 固定 seed 下 smoke、visibility、stress 和 action outcome 的逐 tick 快照；
- Sensor 延迟导致 Agent 使用旧 Observation 时的 reject/accept 行为；
- alarm 与 smoke 同时出现时，Human stress 与 microphone quality 的独立变化；
- manual pilot action 与 Agent action 的冲突裁决；
- shutdown 后 smoke 按 ventilation 衰减，而不是立刻消失；
- event delay、priority 和 subscriber failure 不改变已提交 snapshot 的确定性。

只有因果链、信息链、动作边界和冲突规则都能在 trace/replay 中被解释，才算“各元素交互”被完整建模。
